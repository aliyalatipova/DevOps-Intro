

## Task 1 – Artifact Registries

### 1.1 Сервисы облачных провайдеров

#### AWS – Amazon Elastic Container Registry (ECR)
- **Типы артефактов:** Docker images, OCI images, Helm charts.
- **Ключевые функции:** сканирование уязвимостей (Basic и Enhanced), репликация между регионами, управление доступом через IAM, интеграция с ECS/EKS.
- **Интеграции:** AWS CodeBuild, CodePipeline, CloudFormation.
- **Цены:** плата за объём хранения ($0.10 за ГБ/мес) и передачу данных. Нет платы за вызовы API.
- **Использование:** хранение образов для приложений в Amazon ECS/EKS, CI/CD пайплайны.

#### GCP – Google Artifact Registry
- **Типы артефактов:** контейнеры (Docker), языковые пакеты (npm, Python, Maven, Go, apt, yum и др.).
- **Функции:** георепликация, управление жизненным циклом, автоматическая сборка из Cloud Build, IAM.
- **Интеграции:** Cloud Run, GKE, Cloud Build, Cloud Deploy.
- **Цены:** $0.10 за ГБ/мес (стандартное хранилище) + плата за исходящий трафик.
- **Использование:** универсальный реестр для всех артефактов разработки.


#### Azure – Azure Container Registry (ACR)
- **Типы артефактов:** Docker images, OCI images, Helm charts, OCI артефакты.
- **Функции:** георепликация (Premium), Content Trust, виртуализация реестра, интеграция с Defender for Cloud.
- **Интеграции:** AKS, Service Fabric, App Service, Azure Pipelines.
- **Цены:** три уровня (Basic ~$0.10/ГБ, Standard, Premium) – отличается пропускная способность и функции.
- **Использование:** хранение образов для AKS, CI/CD в Azure DevOps.

### 1.3 Анализ: какой реестр выбрать для мультиоблачной стратегии и почему
*Напишите своими словами, например:*
Для мультиоблачной среды (приложения работают одновременно в AWS и GCP) лучше использовать **независимый от провайдера реестр**, например, Docker Hub, GitLab Container Registry или JFrog Artifactory. Если же строго между этими тремя – GCP Artifact Registry удобнее, так как поддерживает больше типов артефактов (языковые пакеты), что может уменьшить количество инструментов. Но при глубокой привязке к сервисам AWS удобнее ECR. В общем случае каждый провайдер предлагает аналогичный функционал.

## Task 2 – Serverless Computing Platforms

### 2.1 Сервисы провайдеров

#### AWS – Lambda
- **Языки:** Node.js, Python, Java, Go, .NET, Ruby, Rust (через custom runtime).
- **Модели вызова:** событийная (S3, SQS), HTTP (через API Gateway), по расписанию.
- **Холодный старт:** 100–500 мс (зависит от языка, размера пакета).
- **Лимиты:** максимум 15 минут выполнения, 10 ГБ RAM, 50 МБ пакета (или 250 МБ с контейнером).
- **Цены:** $0.20 за 1 млн вызовов, $0.0000166667 за ГБ-секунду.
- **Интеграции:** 200+ сервисов AWS.
- **Использование:** обработка событий, API бэкенды, ETL, чат-боты.


#### GCP – Cloud Functions (2nd gen)
- **Языки:** Node.js, Python, Go, Java, .NET, Ruby, PHP.
- **Триггеры:** HTTP, Cloud Storage, Pub/Sub, Firestore, Firebase.
- **Холодный старт:** 100–300 мс (2nd gen лучше 1st gen).
- **Лимиты:** до 60 минут, до 16 ГБ RAM, размер кода до 100 МБ.
- **Цены:** первые 2 млн вызовов бесплатно, затем $0.40 за 1 млн; $0.0000025 за ГБ-сек (после бесплатного лимита).
- **Интеграции:** всё GCP (BigQuery, Cloud Run, Spanner).
- **Использование:** веб-хуки, обработка файлов, работа с базами данных.


#### Azure – Azure Functions
- **Языки:** C#, JavaScript, Python, Java, PowerShell, TypeScript.
- **Модели:** HTTP, очередь, Cosmos DB, Timer, Event Grid.
- **Холодный старт:** 1–3 секунды (на Consumption плане).
- **Лимиты:** таймаут 10 минут (Consumption) или неограниченно (Premium), память до 1.5 ГБ (Consumption) или 14 ГБ (Premium).
- **Цены:** Consumption – $0.20 за 1 млн выполнений, $0.000016 за ГБ-с.
- **Интеграции:** сервисы Azure, Logic Apps, App Service.
- **Использование:** API, обработка сообщений, интеграционные сценарии.

### 2.3 Анализ: какую платформу выбрать для REST API бэкенда и почему

*Напишите рассуждение, например:*
Для REST API бэкенда с непредсказуемой нагрузкой и требованием низкой задержки лучше всего подходит **AWS Lambda + API Gateway**. Причины: очень низкий холодный старт (особенно с Provisioned Concurrency), огромное количество документации, зрелость экосистемы. Однако GCP Cloud Functions тоже хорош, если вы уже используете GCP, а время старта у него даже чуть лучше. Azure Functions имеет самый высокий холодный старт (~1-3 с), что может быть заметно для пользователей, поэтому её лучше применять для внутренних фоновых задач, а не для публичных API.

### 2.4 Рефлексия: преимущества и недостатки serverless

**Преимущества:**
- Нет управления серверами (ОС, патчи, масштабирование).
- Автоматическое масштабирование от 0 до тысяч.
- Платите только за реальное время выполнения (экономия при редкой нагрузке).
- Быстрая разработка маленьких функций.

**Недостатки:**
- Холодный старт (задержка при первом вызове после простоя).
- Ограничения по времени (максимум 15–60 минут) – не подходит для долгих задач.
- Сложность отладки и мониторинга (требует специальных инструментов).
- Вендор-локин (привязка к конкретному облаку).
